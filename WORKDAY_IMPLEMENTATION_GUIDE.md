# 工作日重置功能实施指南

## 功能概述

本功能实现了"工作日"概念，解决跨天加班的统计问题，并添加了直观的进度条倒计时显示。

### 核心特性

1. **工作日定义**：早上 06:00 - 次日 05:59
2. **跨天处理**：凌晨 00:00-05:59 的提交算前一天
3. **进度条显示**：实时显示当前工作日进度
4. **倒计时功能**：显示距离重置的剩余时间
5. **动态颜色**：根据剩余时间变化颜色

## 工作原理

### 工作日计算逻辑

```typescript
// 凌晨 0:00-5:59 算前一天
if (当前小时 < 6) {
  工作日 = 当前日期 - 1天
} else {
  工作日 = 当前日期
}
```

### 示例场景

#### 场景1：正常工作日
- 2月1日 08:00 提交 → 记录到 2月1日
- 2月1日 18:00 提交 → 记录到 2月1日
- 2月1日 23:00 提交 → 记录到 2月1日

#### 场景2：跨天加班
- 2月1日 22:00 提交 → 记录到 2月1日
- 2月2日 02:00 提交 → 记录到 2月1日 ✅（算前一天）
- 2月2日 06:00 归档 → 归档 2月1日数据
- 2月2日 08:00 提交 → 记录到 2月2日

#### 场景3：凌晨用户体验
- 2月1日 23:50 - 参与人数：5000人
- 2月2日 00:01 - 参与人数：5000人 ✅（不会重置）
- 2月2日 02:00 - 参与人数：5000人 ✅（继续累计）
- 2月2日 06:00 - 参与人数：0人（重置）

```bash

```

### 问题2：数据没有在6点重置

**检查步骤**：
1. 确认定时任务已创建
```sql
SELECT * FROM cron.job WHERE jobname LIKE '%workday%';
```

2. 检查执行日志
```sql
SELECT * FROM cron.job_run_details 
ORDER BY start_time DESC LIMIT 5;
```

**解决方案**：
- 确认 pg_cron 扩展已启用
- 确认时区设置正确
- 手动执行一次归档：`SELECT archive_daily_data();`

### 问题3：凌晨提交的数据归属错误

**检查步骤**：
```sql
-- 检查工作日函数
SELECT get_work_date(NOW());

-- 检查最近的提交记录
SELECT id, user_id, date, submitted_at 
FROM status_records 
ORDER BY submitted_at DESC 
LIMIT 10;
```

**解决方案**：
- 确认 `implement_workday_logic.sql` 已正确执行
- 重新执行脚本

## 注意事项

1. **时区问题**：
   - Supabase 使用 UTC 时区
   - 中国用户需要使用 `setup_workday_archive_cron_beijing.sql`
   - UTC 22:00 = 北京时间次日 06:00

2. **数据完整性**：
   - 原始数据永久保留在 `status_records` 表
   - `daily_history` 只是快照，可以重新生成
   - 归档使用 `ON CONFLICT` 处理，不会重复

3. **性能优化**：
   - 倒计时每分钟更新一次
   - 物化视图需要定期刷新
   - 建议每30分钟刷新一次统计视图

4. **向后兼容**：
   - 现有数据不受影响
   - 可以随时回滚到旧逻辑
   - 历史数据保持不变
